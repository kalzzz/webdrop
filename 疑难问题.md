# 疑难问题记录

## 问题：WebRTC ICE Candidate 无法生成，文件传输卡在 0%

### 现象
- `setLocalDescription` 完成后，`iceGatheringState` 一直是 `new`，不会变成 `gathering`
- `onicecandidate` 事件从未触发，SDP 中没有 `candidate` 行
- 控制台手动测试 WebRTC 正常，但 app 内不正常

### 根因
`createDataChannel` 的调用顺序错误。

**正确顺序（正常）：**
```javascript
pc.createDataChannel('ft');         // 1. 先创建 DataChannel
pc.createOffer().then(o => pc.setLocalDescription(o)); // 2. 再创建 Offer
```

**错误顺序（app 原来）：**
```javascript
pc.createOffer().then(o => pc.setLocalDescription(o)); // 1. 先创建 Offer
pc.createDataChannel('ft');         // 2. 后创建 DataChannel
```

在 Chrome/Chromium 浏览器中，如果 `createDataChannel` 在 `createOffer` 之后调用，部分情况下 ICE 候选收集不会被触发，导致 `iceGatheringState` 卡在 `new` 状态。

### 解决方法
调整 `startTransfer` 函数中 `createDataChannel` 和 `createOffer` 的顺序：

```javascript
// 修改前（错误）
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
const dc = pc.createDataChannel('ft', { ordered: false });  // 在 setLocalDescription 之后

// 修改后（正确）
const dc = pc.createDataChannel('ft', { ordered: false });  // 先创建 DataChannel
const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
```

## 问题：DC open 竞态导致传输不开始

### 现象
- ICE 连接已建立（`iceConnectionState = connected`）
- DataChannel 已打开（`DC open`）
- 但 `sendV2Head` 从未被调用，进度一直是 0%

### 根因
`handleFileAccept` 使用 2 秒轮询等待 DataChannel open，但 DC 在 ICE connected 之后才触发 onopen。轮询 2 秒后退出，而 DC 刚好在退出后变成 open。

### 解决方法
让 `setupDC` 的 `channel.onopen` 回调直接调用 `sendV2Head`，而不是轮询：

```javascript
// 修改前
channel.onopen = () => { /* 只打印日志 */ };

// 修改后
channel.onopen = () => {
    if (role === 'sender') {
        const f = state.outgoingFiles.get(connId);
        if (f) sendV2Head(connId, f);  // 自动触发
    }
};
```

## 问题：V2Head fileSize 解析错误（62PB）

### 现象
- 发送端 `fileSize = 14484155`（14MB）
- 接收端解析为 `62,208,972,035,194,880`（62PB）
- CRC32 校验通过，但数据明显错误

### 排查过程

1. **CRC 通过但数据损坏**
   - 接收端 `recvCrc === calcCrc`，CRC 通过
   - 但 `fileSize bytes at offset 31: 00 00 00 00 00 dd 02 bb`
   - CRC 通过说明两端以同样的方式解析出了同样的错误结果

2. **Node.js 本地模拟重现**
   ```
   发送端写的布局（offset）：
     31: hi = 0             → bytes[31-34] = 00 00 00 00
     35: lo = 14484155      → bytes[35-38] = 00 dd 02 bb

   接收端读的布局（错误）：
     fsLo = readUInt32BE(31) = 0              ← 实际读的是 hi 的值
     fsHi = readUInt32BE(35) = 14484155       ← 实际读的是 lo 的值
     fileSize = 14484155 * 2^32 = 62PB ❌
   ```

3. **找到根因**
   - `sendV2Head` 写：`offset31 = hi`，`offset35 = lo`（big-endian）
   - `recvV2Head` 读：把 `fsLo` 赋给了 offset31（应该是 hi），把 `fsHi` 赋给了 offset35（应该是 lo）
   - **变量名和偏移完全搞反**

### 解决方法
修正 `recvV2Head` 中的偏移：

```javascript
// 修改前（错误）
const fsLo = v.getUint32(o, false);        // 读 offset31 → 实际是 hi
const fsHi = v.getUint32(o + 4, false);   // 读 offset35 → 实际是 lo
const fs = fsLo + fsHi * 0x100000000;

// 修改后（正确）
const fsHi = v.getUint32(o, false);        // 读 offset31 → hi (0)
const fsLo = v.getUint32(o + 4, false);   // 读 offset35 → lo (14484155)
const fs = fsLo + fsHi * 0x100000000;      // = 14484155 ✅
```

## 问题：SCTP 单消息大小限制（4MB 太大）

### 现象
```
[DC error] OperationError: Failure to send data
DC.readyState= closed
```
ICE 是 connected 的，但 `ch.send()` 直接抛出 `OperationError`。

### 根因
`CHUNK_SIZE = 4 * 1024 * 1024`（4MB），而 WebRTC DataChannel 基于 SCTP，单个消息有 16KB 左右的限制。

### 解决方法
```javascript
// 修改前
const CHUNK_SIZE = 4 * 1024 * 1024;  // 4MB

// 修改后
const CHUNK_SIZE = 16 * 1024;  // 16KB
```

## 问题：V2Head 的 `>>>` 优先级 bug

### 现象
`fileSize` 的高 32 位始终等于 `fileSize` 本身，而不是 0。

### 根因
```javascript
// 错误：>>> 优先级高于 /，等价于 Math.floor(fileSize / (0x100000000 >>> 0)) = Math.floor(fileSize / 1)
const hi = Math.floor(fileSize / 0x100000000) >>> 0;

// 正确：先算除法，再无符号右移
const hi = (Math.floor(fileSize / 0x100000000)) >>> 0;
```

## 问题：DC open 时序错误导致未授权即发送文件

### 现象
- 发送端选择文件后，在接收端未点击"接收"的情况下，文件内容就已经开始发送
- 接收端收到文件数据但弹出"接收"对话框还未处理

### 根因
`dc.onopen` 回调中只检查了 `dcReady` 标记，未检查 `file_accept` 是否已收到。DC open 后直接调用 `sendV2Head`，导致文件在接收方同意之前就被发送。

### 设计要求
文件发送必须同时满足两个条件：
1. 接收方已点击"接收"（`file_accept` 已到达）
2. DataChannel 已打开（`DC open`）

**两者缺一不可**。

### 解决方法
引入 `transferAccepted` 标记，只有收到 `file_accept` 时才设为 `true`；`dc.onopen` 必须同时检查 `dcReady` 和 `transferAccepted`：

```javascript
// state 中新增
dcReady: new Map(),        // DC 是否已 open
v2HeadSent: new Map(),     // V2Head 是否已发送
transferAccepted: new Map() // 是否已收到 file_accept

// dc.onopen（发送端）
dc.onopen = () => {
    state.dcReady.set(connId, true);
    // 必须同时满足：DC open + 已收到 file_accept
    const ch = state.dataChannels.get(connId);
    if (ch && ch.readyState === 'open' && state.transferAccepted.get(connId) && !state.v2HeadSent.get(connId)) {
        const f = state.outgoingFiles.get(connId);
        if (f) sendV2Head(connId, f);
    }
};

// handleFileAccept（发送端）
function handleFileAccept(msg) {
    state.transferAccepted.set(connId, true);  // 先设标记
    if (state.dcReady.get(connId)) {
        const ch = state.dataChannels.get(connId);
        if (ch && ch.readyState === 'open' && !state.v2HeadSent.get(connId)) {
            const f = state.outgoingFiles.get(connId);
            if (f) sendV2Head(connId, f);
        }
    }
    // 若 DC 尚未 open，dc.onopen 会检查 transferAccepted 后处理
}
```

### 清理
传输终止时（reject / failed / 结束）需同时清理这三个标记：

```javascript
// handleFileReject / cleanup / removeTransferUI 中
state.dcReady.delete(connId);
state.v2HeadSent.delete(connId);
state.transferAccepted.delete(connId);
```

## 教训

1. **WebRTC 操作顺序敏感**：`createDataChannel` 必须在 `createOffer` 之前
2. **协议设计要打印 hex**：通过打印原始字节的 hex 值，能直观看出字节布局是否正确
3. **Node.js 可独立重现问题**：不需要两个浏览器页面，直接在 Node.js 中模拟编解码能快速定位 bug
4. **CRC 通过但数据损坏说明 bug 在解析逻辑**：不是传输层的问题，而是对协议格式的理解错误
5. **涉及字节序的协议要逐字节确认**：send 和 recv 两端的偏移必须严格对照
6. **异步事件时序要用状态机思维**：DC open 和 file_accept 到达顺序不确定，用标记而不要假设先后
